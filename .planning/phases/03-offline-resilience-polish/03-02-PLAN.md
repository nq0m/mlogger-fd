<objective>
Deliver offline dupe checking: when the server is unreachable, the QSO entry form checks for duplicates against locally cached QSO history in IndexedDB. The cache is populated on page load with the full QSO history from the server and kept current via WebSocket events. Per D-07, locally queued (unsynced) QSOs are included in the dupe check.

Purpose: Without offline dupe checking, operators working offline could repeatedly log the same station — invalidating contest scores. This closes the last gap in the offline logging flow.
Output: Dupe warnings appear offline; IndexedDB cache mirrors server state
</objective>

1. **frontend/src/lib/db.js**: Add `cached_qsos` table to the Dexie schema:
   - Update `db.version(1).stores({...})` to include `cached_qsos: 'client_id, [callsign+band+mode], timestamp'`
   - If db.js already has version(1) locked, bump to `version(2).stores({ cached_qsos: 'client_id, [callsign+band+mode], timestamp' })` — Dexie auto-upgrades version on schema change
   - Add export function `async loadCachedQsos()`: returns `db.cached_qsos.orderBy('timestamp').reverse().toArray()`
   - Add export function `async populateCache(qsos)`: calls `db.cached_qsos.bulkPut(qsos)` (bulkPut upserts — same client_id overwrites, preventing duplicates). Each qso object shape: `{ client_id, id, timestamp, callsign, band, mode, recv_exchange, operator, is_dupe, points }`
   - Add export function `async addToCache(qso)`: calls `db.cached_qsos.put(qso)` (single upsert for WebSocket events)
   - Add export function `async offlineDupeCheck(callsign, band, mode)`: 
     - Query `db.cached_qsos.where({callsign: callsign.toUpperCase(), band, mode}).count()` — exact match per D-06
     - ALSO query `db.queued_qsos` by checking each queued QSO's `qso.callsign`, `qso.band`, `qso.mode` match the input (Dexie can't do compound index on nested JSON — iterate `db.queued_qsos.toArray()` and filter in JS since queue is small, typically < 50 items). Per D-07: queued QSOs must be included.
     - If count from cached_qsos > 0 OR any queued QSO matches: return `{ is_dupe: true }`
     - Otherwise return `{ is_dupe: false }`
     - Note: offline dupe does NOT check similar calls (D-06: partial call similarity remains server-side, online-only). Return shape does not include `similar_calls`.

2. **frontend/src/lib/stores/qso.svelte.js**: Add cache population on app load (D-08):

     - Fetches `GET /api/qso?limit=9999` (full QSO history — ~3000 QSOs max per D-05)
     - On success, calls `await populateCache(data)` — stores all QSOs in IndexedDB cached_qsos
     - On failure (network error — server unreachable at load time), silently skip — cache population will happen when API becomes available (or on next load)
   - Call `loadCache()` at module top-level (similar to `refreshQueueCount()` in sync.svelte.js) so it fires on first import

Simplest: if no `client_id`, generate one with `crypto.randomUUID()` so the cache entry has a unique key.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| IndexedDB cached_qsos ↔ QsoEntryForm | Dupe check reads from client-side cache — data integrity depends on cache being populated from server |
| GET /api/qso?limit=9999 → IndexedDB | Cache source is server data — trusted boundary; server is authority |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-03 | Tampering | IndexedDB cached_qsos — offline dupe bypass via console | accept | Client-side dupe check is advisory (dupes are still logged as dupes with 0 points). Server-side dupe check on sync re-validates. Duplicate QSOs inserted offline will be marked as dupes when synced. |
| T-03-04 | Information Disclosure | QSO history in IndexedDB | accept | QSO cache mirrors server data already available on the LAN. No PII — callsigns, band/mode pairs are public information by FCC regulation. No authentication on LAN per AGENTS.md. |
</threat_model>

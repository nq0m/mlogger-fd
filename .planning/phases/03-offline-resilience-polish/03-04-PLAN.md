**UX-04 — per RESEARCH.md § Pattern 2 (Built-in SW, no Workbox):**

1. **frontend/src/service-worker.js** (NEW FILE): Create at the SvelteKit convention path `frontend/src/service-worker.js`. Copy the implementation from RESEARCH.md § Pattern 2 (lines 226-261) with modifications:
   
   - Use `/// <reference types="@sveltejs/kit" />` at top
   - Import `{ build, files, version }` from `'$service-worker'`
   - `CACHE = \`fdlogger-\${version}\`` for cache versioning via hashed filenames
   - `ASSETS = [...build, ...files]` — all Vite-generated assets + static files
   - `install` event: `event.waitUntil(caches.open(CACHE).then(cache => cache.addAll(ASSETS)))` — pre-caches app shell
   - `activate` event: `event.waitUntil(caches.keys().then(keys => Promise.all(keys.filter(k => k !== CACHE).map(k => caches.delete(k)))))` — cleans old cache versions
   - `fetch` event: 
     - If `event.request.method !== 'GET'` — return (don't cache POST/PUT)
     - If `url.pathname.startsWith('/api/')` — return (per Pitfall 1: no caching of API responses, ever)
     - Otherwise: `event.respondWith(caches.match(event.request).then(cached => cached || fetch(event.request)))` — cache-first for app shell
   
   Per RESEARCH.md Open Questions § 2 recommendation: add `self.skipWaiting()` at the START of the `install` event handler (before cache.addAll) to activate new SW immediately. This is safe in a LAN deployment with single server version.

2. **SvelteKit registration:** SvelteKit automatically registers the service worker in production builds via `$service-worker`. No manual registration code needed in `app.html`. The `adapter-static` with `fallback: 'index.html'` (already configured in svelte.config.js) ensures the SW file ends up in the build output.

3. **Verify SW is in build output:** After `npm run build`, confirm `frontend/build/service-worker.js` exists. SvelteKit's Vite build auto-bundles `src/service-worker.js` → `build/service-worker.js`. The Go `embed.FS` serves it as a static file.

4. **No changes to svelte.config.js needed** — static adapter with fallback is already configured correctly for SW support.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Service Worker fetch → /api/* | SW intercepts all fetch requests — must not cache API responses |
| Browser cache → SPA shell | SW serves cached index.html + assets — must invalidate on deploy (versioned cache key) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-07 | Spoofing | Service Worker stale cache | mitigate | Cache key includes `version` from `$service-worker` (hashed filenames from Vite build). New deploy produces new version → old cache evicted in `activate`. `self.skipWaiting()` ensures immediate activation. |
| T-03-08 | Information Disclosure | Service Worker caching /api/* responses | mitigate | Fetch handler explicitly skips caching for URLs starting with `/api/` (early return). API data is never stored in Cache Storage. |
| T-03-09 | Denial of Service | Debounce cooldown blocking legitimate rapid entry | accept | 1000ms cooldown is intentionally chosen as the floor for Field Day logging speed. Accept risk: an operator logging faster than 1 QSO/second may hit the guard — this is rare in practice and consistent with contest logging norms. |
</threat_model>

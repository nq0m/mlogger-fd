<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client→POST /api/sync | Untrusted client sends batch QSO array over LAN — must validate each QSO and dedup by client_id |
| IndexedDB↔Browser | Client-side storage accessible via browser console — data is non-authoritative until server validates |
| Queue state ($state) | In-memory queue count — informational only, server is authority |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | POST /api/sync handler | mitigate | `model.ValidateRequired()` validates callsign/band/mode/recv_exchange per QSO; batch rejected with 400 if any QSO fails. `client_id TEXT UNIQUE` with `ON CONFLICT DO NOTHING` prevents duplicate insertion from replayed batches. |
| T-03-02 | Tampering | IndexedDB queued_qsos | mitigate | Server-side validation on sync (T-03-01) ensures tampered IndexedDB data is rejected or sanitized. Invalid QSOs in queue result in server 400 — client should log and clear on 400 (future enhancement). |
| T-03-03 | Spoofing | Queue count display | accept | Queue count is derived from IndexedDB record count — purely informational. Tampering only affects the displayed number, not data integrity. |
| T-03-SC | Tampering | npm install dexie | mitigate | Blocking human checkpoint (Task 1) verifies dexie package legitimacy at npmjs.com before install. Package age 8+ years, 2M+ weekly downloads, verified source repo github.com/dexie/Dexie.js. |
</threat_model>

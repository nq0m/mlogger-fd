  <action>
Create internal/ws/hub.go with package ws.

**Hub struct** — fields: mu sync.RWMutex, clients map[*Client]bool, broadcast chan []byte (buffer 256), register chan *Client, unregister chan *Client.

**Client struct** — fields: hub *Hub, conn *websocket.Conn, send chan []byte (buffer 64).

**NewHub()** — returns &Hub with initialized channels and map.

**Hub.Run()** — goroutine that loops on select over register/unregister/broadcast channels. On register: h.mu.Lock(); h.clients[client] = true; h.mu.Unlock(); slog.Info. On unregister: h.mu.Lock(); if exists, delete and close(client.send); h.mu.Unlock(); slog.Info. On broadcast message: h.mu.RLock(); for each client, non-blocking select { case client.send <- message: default: close(client.send); delete from map }. h.mu.RUnlock().

**Hub.Broadcast(v interface{}) error** — json.Marshal v then push to h.broadcast chan. Return error on marshal failure.

**Client.ReadPump()** — reads from conn in a loop, exits on error. Sets ReadDeadline with 60s pong wait. On exit, sends client to hub.unregister and closes conn.

**Client.WritePump()** — reads from send channel, writes to conn. Exits when channel closed. Writes ping messages periodically.

Follow RESEARCH.md Pattern 1 exactly for the Hub implementation. Follow RESEARCH.md anti-patterns: do NOT hold hub mutex during WS writes (use per-client buffered send channels); do NOT iterate clients directly from handler goroutines (always go through Hub.broadcast channel).

Create internal/ws/hub_test.go. Use gorilla/websocket for test WebSocket connections (dial test server). Create a small httptest server that upgrades connections, register/unregister clients, test broadcast fan-out. Test with -race flag.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → Server /ws | WebSocket upgrade crosses LAN trust boundary — any LAN client can open a WebSocket |
| Client → Server POST /api/qso | HTTP POST crosses same LAN trust boundary — any LAN client can submit QSOs |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-SC | Tampering | go get github.com/gorilla/websocket | mitigate | Human checkpoint (Task 1) verifies package on pkg.go.dev — confirms 44K+ importers, BSD-2-Clause license, no slopcheck flags |
| T-02-05 | Spoofing | WebSocket upgrade handler (ServeWS) | accept | CheckOrigin returns true for all origins (trusted LAN per AGENTS.md). If open WiFi vector added later, replace with Origin validation against known LAN IP range and/or shared-secret token check. |
| T-02-06 | Denial of Service | WebSocket frame parsing | mitigate | gorilla/websocket handles RFC 6455 frame parsing (Autobahn test suite verified). chi middleware.Recoverer catches panics. Set default ReadBufferSize/WriteBufferSize (1024) to prevent excessive memory allocation. Hub broadcast channel buffered (256) to absorb QSO bursts. |
| T-02-07 | Denial of Service | Hub broadcast loop blocked by slow client | mitigate | Per-client buffered send channel (64) with non-blocking select/default — slow clients' messages are dropped rather than blocking the hub. Dropped clients are unregistered. |
| T-02-08 | Denial of Service | Goroutine leak from WebSocket connections | mitigate | ReadPump exits when ReadMessage returns error (connection close). defer unregisters client and closes conn. SetReadDeadline(60s) via pong handler detects zombie connections. |
| T-02-09 | Tampering | JSON injection in WebSocket messages | accept | All WebSocket messages are server-generated via json.Marshal. Client receives via JSON.parse() — safe in browsers. No user input is eval'd. |
</threat_model>
